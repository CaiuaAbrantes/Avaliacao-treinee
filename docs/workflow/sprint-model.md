
# Sprint Model

## Objetivo

O desenvolvimento do projeto será organizado utilizando ciclos curtos de trabalho (Sprints). Cada Sprint possui objetivos bem definidos, permitindo acompanhar a evolução do sistema, distribuir as atividades entre os integrantes e validar continuamente as funcionalidades implementadas.

Cada Sprint resulta em uma entrega parcial do projeto, possibilitando integração frequente entre os componentes desenvolvidos e facilitando a identificação de problemas durante o desenvolvimento.

---

# Ciclo de uma Sprint

Cada Sprint seguirá o seguinte fluxo de trabalho:

1. Planejamento das atividades.
2. Distribuição das tarefas entre os integrantes.
3. Desenvolvimento das funcionalidades previstas.
4. Integração entre os componentes desenvolvidos.
5. Testes e validação das funcionalidades implementadas.
6. Revisão da Sprint.
7. Atualização da documentação e preparação da próxima Sprint.

---

# Estrutura de uma Sprint

Cada Sprint será documentada contendo as seguintes informações:

* Objetivo da Sprint;
* Período de execução;
* Responsáveis pelas atividades;
* Funcionalidades planejadas;
* Funcionalidades concluídas;
* Dificuldades encontradas;
* Resultados obtidos.

---

# Controle das atividades

Durante a execução das Sprints, a equipe utilizará ferramentas de acompanhamento, como Kanban e diagrama de Gantt, permitindo visualizar:

* tarefas pendentes;
* tarefas em desenvolvimento;
* tarefas concluídas;
* responsáveis por cada atividade;
* cronograma de execução.

---

# Critérios para conclusão de uma Sprint

Uma Sprint será considerada concluída quando:

* todas as funcionalidades previstas forem implementadas;
* os testes planejados forem executados;
* a integração entre os componentes estiver validada;
* a documentação correspondente for atualizada;
* as alterações forem registradas no repositório Git por meio de commits.

---

# Relação entre Sprints e Versionamento

Ao término de cada Sprint, uma nova versão do projeto poderá ser disponibilizada seguindo o planejamento de versionamento semântico adotado pela equipe.

Essa estratégia permite acompanhar simultaneamente a evolução do cronograma, das funcionalidades implementadas e das versões entregues, garantindo maior organização durante todo o desenvolvimento do projeto.
